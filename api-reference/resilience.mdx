## Module entrypoints

- `bytekit/retry-policy`
- `bytekit/request-cache`
- `bytekit/request-deduplicator`
- `bytekit/rate-limiter`
- `bytekit/error-boundary`

## `RetryPolicy`

Encapsulates retry behavior for transient failures.

### Config

| Option | Purpose |
| --- | --- |
| `maxAttempts` | Maximum retry attempts |
| `initialDelayMs` | Starting delay between attempts |
| `maxDelayMs` | Delay cap |
| `backoffMultiplier` | Exponential growth factor |
| `shouldRetry` | Custom retry predicate |

### Main method

- `execute(fn)` runs an async operation with the configured policy.

## `CircuitBreaker`

Protects your app from repeatedly calling an unstable dependency.

### Config

| Option | Purpose |
| --- | --- |
| `failureThreshold` | Open after this many failures |
| `successThreshold` | Close again after enough successful probes |
| `timeoutMs` | Wait time before half-open retry |
| `errorMessageFormatter` | Custom open-circuit error message |

### Main methods

- `execute(fn)`
- `getState()`
- `reset()`

## `RequestCache`

Simple cache for request results.

### Config

| Option | Purpose |
| --- | --- |
| `ttl` | Fresh lifetime |
| `staleWhileRevalidate` | Stale grace period |
| `keyGenerator` | Custom key strategy |

### Main methods

- `get(url, options?)`
- `set(url, data, ttl?, options?)`
- `isStale(url, options?)`
- `invalidate(url, options?)`
- `invalidatePattern(pattern)`
- `clear()`
- `getStats()`
- `prune()`

## `RequestDeduplicator`

Shares the same in-flight promise across equivalent requests.

### Main methods

- `execute(url, fn, options?)`
- `getStats()`
- `getInFlightCount()`
- `clear()`
- `resetStats()`

## `RateLimiter`

Token bucket limiter for outbound request control.

### Main methods

- `isAllowed(url)`
- `getStats(url)`
- `waitForAllowance(url)`
- `reset(url)`
- `resetAll()`

## `SlidingWindowRateLimiter`

Alternative limiter with tighter accuracy and more memory usage.

It exposes the same main control methods as `RateLimiter`.

## `ErrorBoundary`

Use this module when you need centralized error normalization or safer boundaries around failure-prone code paths.

<Tip>
  In day-to-day application code, many teams combine `ApiClient`, `RetryPolicy`, `CircuitBreaker`, `RequestCache`, and `RateLimiter` rather than using just one primitive.
</Tip>
